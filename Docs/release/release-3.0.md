# Release 3.0

建议 tag: `v3.0.0`

## 当前进展

1. 前端完成 V3.0 封板，hero 与 chat 共用一套统一交互界面，整体体验已经稳定可用。
2. 模型切换能力已经接入现有流式对话链路，默认值读取 `.env`，用户选择保存在浏览器本地，并从下一条对话开始生效。
3. settings 组件完成多轮视觉与交互收敛，已从常驻卡片优化为极简三横入口，展开后提供模型选择，选定后自动收起。
4. 模型列表不再依赖浏览器原生下拉，而是改为自绘列表，交互高亮、阴影和输入框风格统一，视觉一致性显著提升。
5. settings 在 hero / chat 两种视图下的位置已统一对齐，解决了状态切换时横向跳动与纵向不一致的问题。

## 本次重点解决的问题

1. 解决了前端无法直接消费 `AVAILABLE_MODELS` 的问题，通过 `Gateway/Front/app.py` 把模型列表与当前默认模型注入到前端配置。
2. 解决了模型选择和后端推理之间的联动问题，前端请求现在会透传 `model`，无需修改现有流式协议主体。
3. 解决了“切换模型何时生效”的产品语义问题，当前规则明确为“从下一条新消息开始生效”，且不打断正在进行中的流式响应。
4. 解决了 settings 视觉过重、原生下拉风格割裂、hero/chat 位置不一致等前端体验问题。
5. 解决了用户刷新页面后模型选择丢失的问题，当前通过本地持久化保证使用连续性。

## 后续问题与下一阶段关注点

1. 移动端适配：需要继续检查小屏布局、触控交互和安全区域，确保 settings、输入框、消息流在手机端保持稳定体验。
2. 部署发布上云：需要梳理前后端启动方式、环境变量注入方式、静态资源组织方式，以及发布后的可观测性与回滚策略。
3. 约束前端通信规范：需要明确前端与 orchestrator / backend 之间的请求体、事件流、错误码、状态事件和兼容策略，减少后续扩展时的耦合与歧义。
4. 为 skill 进场提供良好环境：在 UI、协议和状态管理层预留稳定接口，为后续 skill 编排、工具回传、结构化结果展示打基础。

## 发布建议

1. 本次发布适合作为前端交互封板版本，优先冻结 UI 主路径，避免继续在基础交互上频繁摆动。
2. 发版后建议把精力转向移动端、云上部署和通信规范三条主线，减少纯视觉层面的返工成本。
3. 如果后续进入 skill 集成阶段，建议先定义协议与容器，再落具体 skill，以免重复调整前端骨架。
